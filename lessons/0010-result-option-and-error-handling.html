Lesson 10: Result, Option, and the ? Operator

No exceptions in Rust — failure is a value, and the compiler makes you handle it.

Why this matters

Python signals failure by raising an exception that unwinds the stack until something catches it — or nothing does, and the program crashes. Rust has no exceptions for ordinary errors. A function that might fail returns Result<T, E>, and the compiler forces you to acknowledge that every time you call it. This is a real design difference, not just syntax. Lesson 9's HashMap::get already returned an Option without stopping to explain it — this lesson covers Option properly, alongside its error-carrying sibling Result.

Field study first

A service like this might resolve one operating parameter into a typed value, where the parameter can legitimately be absent and the raw value can be malformed: 

pub fn resolve_heave_limit(
    parameters: &HashMap<String, String>,
    site: &AssetId,
) -> Result<Option<f64>> {
    let key = format!("heave_limit.{site}");

    let Some(raw) = parameters.get(&key) else {
        return Ok(None);
    };

    let limit: f64 = raw
        .trim()
        .parse()
        .context("invalid heave_limit value")?;

    Ok(Some(limit))
}

Two different kinds of "no answer" live in this one signature: None means the parameter simply wasn't set for this site (not an error), while the ? after .context(...) means the parameter was set but couldn't be parsed as a number (a real error, with a message attached). Without ?, that last line would need a manual match to propagate the parse failure.

The theory

  • Result<T, E> is an enum with two variants: Ok(T) (success, holding the value) and Err(E) (failure, holding the error). It's the same enum-with-data mechanism from Lesson 4 — Result isn't special syntax, it's a real enum you could have written yourself.
  • Option<T> is Rust's answer to "might be absent" — Some(T) or None. Python uses None for both "absent" and, informally, "something went wrong." Rust splits these into two distinct types on purpose: a function returning Option<User> means "might not find one"; a function returning Result<User, Error> means "might fail, and here's why."
  • The ? operator unwraps an Ok/Some and keeps going, or immediately returns the Err/None from the whole function. It only works inside a function whose own return type is compatible.
  • let ... else { ... } (the let-else in the snippet above) is a pattern-match that either succeeds and binds the value, or falls through the else branch — here, that branch returns Ok(None) immediately, short-circuiting the rest of the function. It's a clean way to say "handle the absent case up front, then move on assuming the value is there."
  • anyhow vs thiserror: a hand-written enum like Lesson 4's AccessOutcome is the shape you reach for with thiserror when callers need to react differently to different failures — specific, matchable variants. anyhow::Result (used here) is for application code that just needs to report a good error message, without callers needing to match on error type. .context("...") attaches a human-readable note to whatever error occurred, without changing its type.

    Write a function using ? and .context()

    Add this scaffold to src/lib.rs:

    use anyhow::{Context, Result};

/// Parses a raw heave-limit string into a floating-point value.
fn parse_heave_limit(raw: &str) -> Result<f64> {
    todo!()
}

#[test]
fn parse_heave_limit_parses_valid_input() {
    assert_eq!(parse_heave_limit("2.5").unwrap(), 2.5);
}

#[test]
fn parse_heave_limit_errors_on_invalid_input() {
    assert!(parse_heave_limit("not a number").is_err());
}

    Write the body yourself. You'll need str::parse::<f64>() (which returns a Result) and ?, in the same shape as the field study's raw.trim().parse()?. Use .context("...") on the call, the same way resolve_heave_limit does above. Get both tests green.

    Then add a small Option-returning lookup alongside it — no Result needed here, since a missing override isn't an error: 

    use std::collections::HashMap;

/// Looks up a raw heave-limit override by key, if one was set.
fn lookup_heave_limit_override(parameters: &HashMap<String, String>, key: &str) -> Option<String> {
    todo!()
}

#[test]
fn lookup_heave_limit_override_returns_none_when_missing() {
    let parameters = HashMap::new();
    assert_eq!(lookup_heave_limit_override(&parameters, "heave_limit.site-1"), None);
}

#[test]
fn lookup_heave_limit_override_returns_value_when_present() {
    let mut parameters = HashMap::new();
    parameters.insert("heave_limit.site-1".to_string(), "3.2".to_string());
    assert_eq!(
        lookup_heave_limit_override(&parameters, "heave_limit.site-1"),
        Some("3.2".to_string())
    );
}

    This one's a one-liner: .get(key) plus .cloned() — the exact HashMap::get-returns-Option behavior from Lesson 9's checkpoint, now put to use. Get both new tests green too.

Tangible win A function that fails cleanly on bad input instead of panicking, plus one that reports "not set" without treating it as an error — the exact ? + .context() idiom the field study's resolve_heave_limit uses.

Why this matters for the backend

Every module handler in a real backend service tends to follow this exact shape: fn ... -> Result<...>, littered with ?. You can now read the control flow of any of them without getting lost in error-handling noise.
