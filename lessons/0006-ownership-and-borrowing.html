Lesson 6: Ownership & Borrowing

The concept most Python developers find genuinely new about Rust — and the reason that unexplained & from Lesson 4 existed.

Why this matters

Python never makes you think about who owns a value — the garbage collector cleans up whenever nothing references it anymore. Rust has no garbage collector. Instead, every value has exactly one owner, and the compiler tracks this at compile time. This is what the borrow checker is actually checking. Once this clicks, most "why won't this compile" moments in a real backend stop being mysterious.

Field study first

A service like this might evaluate a transfer request and log an audit entry from it. Open your editor and read this method: 

pub fn evaluate(&self, request: TransferRequest) -> AccessOutcome {
    self.record_audit_entry(request.clone());
    if !self.window_is_open() {
        return AccessOutcome::Blocked("outside access window".to_string());
    }
    self.decide(request)
}

record_audit_entry and decide both take request: TransferRequest — by value, not by reference. Ask yourself: why does the first call need request.clone() but the second call (the last line) just uses request plain?

Vim reminder — useful for this exact question * — jump to the next use of the word under the cursor

* searches forward for the word under the cursor, whole-word — not a mnemonic letter, just a motion worth knowing. Put the cursor on request anywhere in this function and hit * to jump straight to its next use, tracing exactly where a value gets consumed versus where it's still needed. Faster than scanning by eye for a short function like this one, and essential once field studies get longer.

The theory

  • Every value has exactly one owner. When you pass a value to a function by value (no &), ownership moves into that function. The original variable can no longer be used — not "might be stale," genuinely will not compile if you try.
  • Borrowing (&) lets you use a value without taking ownership. A function taking &TransferRequest instead of TransferRequest only looks at the value; the caller keeps ownership and can use it again afterward.
  • .clone() makes a second, independent owned copy when you genuinely need to use a value more than once in a context that takes ownership each time. It's not free (it copies data), but it's simple, explicit, and often the right call — exactly what evaluate does above, because it needs request twice and record_audit_entry/decide both take ownership.
  • One small piece of syntax for the exercise below: (value1, value2) is a tuple literal — a fixed-size, ordered grouping of values, possibly of different types, with no field names. (String, String) as a return type means "returns exactly two Strings, in this order"; the caller reads them back out positionally, e.g. result.0/result.1, or destructures them with let (a, b) = ....

Answer to the field-study question: the first call needs .clone() because after it, the original request is still needed for the possible second call on the last line — without cloning, the first call would have moved request away, and that last line wouldn't compile.

Why this matters for the backend

Once you can spot "this takes ownership, this borrows, that's why there's a .clone() here" on sight, reading any backend file gets meaningfully easier — this pattern repeats constantly across a real codebase.
