Lesson 8: Navigating History with jj

jj edit — going back to fix an earlier change without disturbing the ones on top of it.

Why this matters

Lesson 3 gave you describe/new — enough to move forward. Real work isn't linear: you'll spot a mistake in something you already described, want to fix it, and not want to lose the changes you made after it. That's what jj edit is for, and it's exactly the situation you'll hit constantly in a real review cycle on a backend service. Also a deliberate breather after two dense Rust lessons in a row (ownership, then lifetimes).

The idea

jj edit <change-id> moves your working copy to point at any existing change, not just the tip. Whatever you do next edits that change directly — and because jj tracks changes by a stable ID rather than by position, everything built on top of it gets automatically rebased once you're done. You don't rewrite history by hand; jj does it for you.

Today's tasks

  1. Find something worth fixing

    In rust-practice:

    jj log

    Find the change from Lessons 1-2 — per Lesson 3, that's the single change containing both the add and max functions, since the underlying git repo had zero commits until you described it there. Note its change ID — the short alphanumeric string on the left of each log line, not the commit hash.

  2. Vim reminder — before you start this one A — append at the end of the line, no need to navigate there first

    You're appending a clause to the end of an existing doc comment, not rewriting it — A (mnemonic: Append at end of line) drops you into insert mode right after the last character on the line, no need to navigate there with $ first.

    Edit it directly 
    jj edit <that-change-id>

    Your working copy now shows exactly the state of the repo at that point in history. Fix something small and real in it — for example, tighten the doc comment on add to also mention it panics on integer overflow in debug builds (true, and worth knowing: Rust checks for overflow and panics by default in debug builds; the release profile defaults overflow-checks to false, i.e. wrapping, but that's a Cargo profile setting in Cargo.toml, not a language guarantee — a project can and some do turn overflow checks on for release too).

  3. Watch the rebase happen 
    jj log

    Everything that was built on top of the change you just edited — Lesson 4's AccessOutcome, Lesson 5's AccessRun/ready_count, Lesson 6's both_versions, Lesson 7's resolve_access_label, whatever else is there — should still be present, now rebased on top of your edit. Nothing was lost; you didn't have to redo anything by hand.

  4. Get back to your actual current work

    Find the change ID that was your tip before you started (still visible near the top of jj log), and: 

    jj edit <tip-change-id>

    Or, if you'd rather start something fresh from here: jj new.

Why this is safe to experiment with

If anything about this feels wrong afterward — descendants look broken, you edited the wrong change — jj undo reverts the last operation, full stop. There's no "detached HEAD" danger here the way there is in git; jj doesn't have an equivalent failure mode for this operation.

Tangible win You changed something in the middle of your history and watched everything after it update automatically — no manual rebase, no lost work.

Why this matters for the backend

Review feedback on a real backend-service change almost never targets your latest commit only — you'll be asked to fix something two changes back while later work already exists on top. This is that exact workflow, practiced now instead of improvised under review pressure.

Ask your tutor

If the descendants don't look right after editing, or you're not sure which change ID is which, stop and ask before experimenting further — jj is safe to recover from mistakes in, but it's still worth understanding what happened rather than just moving past confusion.
