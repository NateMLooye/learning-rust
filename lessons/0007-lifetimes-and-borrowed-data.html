Lesson 7: Lifetimes — Naming How Long a Borrow Lasts

Lesson 6 taught you what borrowing is. This is how the compiler tracks how long a borrow stays valid — usually invisibly, sometimes not.

Why this matters

You've already been reading and writing lifetimes without knowing it — every & you wrote in Lesson 6 has one. Most of the time the compiler infers it silently. This lesson is about the much smaller set of cases where it can't, and you have to say what you mean. Once this clicks, compiler messages that mention "lifetime" or "borrowed value does not live long enough" stop being mysterious — they're the exact same ownership model from Lesson 6, just applied to how long instead of who owns.

Field study first — a lifetime you've already read, then one you haven't

Lesson 5's struct field study showed you domain structs paired with small impl blocks. A service like this might give one of those structs a method like this: 

pub struct AssetRecord {
    name: String,
}

impl AssetRecord {
    /// Returns this asset's human-readable label.
    pub fn label(&self) -> &str {
        &self.name
    }
}

label takes a reference (&self) and returns a reference (&str) — and has zero lifetime annotations written by hand. Now a case where explicit naming isn't optional — a free function, no self, choosing between two borrowed site labels: 

fn pick_longer_label<'site>(site_a: &'site str, site_b: &'site str) -> &'site str {
    if site_a.len() >= site_b.len() {
        site_a
    } else {
        site_b
    }
}

Two input references, one output reference — and this time, 'site is written out on all three. Same shape as label (reference in, reference out), but this one doesn't compile without the annotation. The theory section explains exactly why.

The theory

  • Every reference has a lifetime — the scope for which it's valid. This isn't new; it's been true of every & since Lesson 6. What's new is that the compiler sometimes needs you to name that scope so it can check a relationship between two or more references.
  • Lifetime elision has rules, not vibes. The Rust compiler applies these in order: (1) every reference parameter gets its own lifetime; (2) if there's exactly one input lifetime, the output gets that same one; (3) if one of the parameters is &self/&mut self, the output gets self's lifetime. label(&self) -> &str is really fn label<'a>(&'a self) -> &'a str — rule 3 fills it in for you, silently.
  • pick_longer_label fits none of the elision rules. It has two input references (site_a, site_b) and isn't a method — no self. Rule 2 needs exactly one input lifetime; rule 3 needs a self. Neither applies, so the compiler has no way to know which input the output reference is tied to — you have to say so.
  • A lifetime annotation is a promise you're stating, not a request you're making. Writing 'site on three places doesn't make anything live longer — Rust doesn't extend lifetimes to satisfy annotations. It tells the compiler "this output is only ever valid as long as these specific inputs are," so the compiler can reject any caller that would violate that. Same enforcement spirit as Lesson 6's move checker, applied to duration instead of ownership.
  • Meaningful names beat 'a. 'site tells a reader what the lifetime represents (a borrowed site label) the same way a good variable name does — 'a is the generic fallback for when nothing more specific is available, not the default style to reach for.
Vim reminder — before you start this one ci< — change just the lifetime inside the angle brackets

Same change + inner pattern from Lesson 1's ci{, different text object: the <'site> parameter list is an angle-bracket pair, so ci< on pick_longer_label<'site> changes just 'site, brackets left in place. Useful the moment you need to rename a lifetime consistently — including while writing your own resolve_access_label<'source> below.

  2. Write a function the elision rules can't handle — full signature yourself

    No stub, no signature this time. The failing tests come first, before you design anything against them:

    Write a function resolve_access_label that takes a primary and a fallback string slice, and returns primary if it's non-empty, otherwise fallback. 
    #[test]
fn resolve_access_label_prefers_primary_when_present() {
    assert_eq!(resolve_access_label("primary", "fallback"), "primary");
}

#[test]
fn resolve_access_label_falls_back_when_primary_empty() {
    assert_eq!(resolve_access_label("", "fallback"), "fallback");
}

    This is the same deliberate scaffold fading from Lesson 2's max, one level further: last time the parameter/return types were already obvious from the test; this time you also have to decide the lifetime parameter itself. Write the full signature and body yourself — no stub. This has the exact shape that breaks elision: two &str inputs, no self, one &str output that could come from either parameter. You'll need .is_empty() and an explicit lifetime parameter on the signature. Get both tests green.

Optional: trigger E0106 on purpose

Before adding the lifetime, try writing the signature with no annotation at all: fn resolve_access_label(primary: &str, fallback: &str) -> &str. Read the "missing lifetime specifier" error — it will name the exact ambiguity from the theory section (which input does the output borrow from?) rather than a generic complaint. Then add <'source> and watch it resolve.

Tangible win A function you designed end-to-end that only compiles because you told the compiler the relationship it couldn't infer on its own — the same gap pick_longer_label closes above.

Why this matters for the backend

Anywhere a backend service resolves a smaller piece out of a larger borrowed structure — picking one label out of a request without cloning the whole request just to hand back a reference to one field of it — this pattern is how it avoids unnecessary allocation. You now know why those signatures carry a named lifetime instead of nothing.

Answer from memory before opening the reveal:

  1. What does the compiler do if a match on an enum leaves out a variant?
  2. You pass a String into a function by value (no &). Can the caller still use that variable afterward?
Reveal the answers

1. It refuses to compile — match is exhaustive, a forgotten variant is a compile error, not a silently-skipped branch (Lesson 4).

2. No. Ownership moved into the function; the original variable is gone, and using it again is a compile error — not "might be stale," won't build (Lesson 6).
