Lesson 14: Iterators & Closures

One synthetic line validates an entire list — using the trait you designed in Lesson 11.

Why this matters

Python's list comprehensions and map/filter have direct Rust equivalents, but Rust's iterator methods chain together and, once the optimizer is turned on (a release build, or opt-level set in a debug profile), typically compile down to the same code as a hand-written loop — "zero-cost abstraction" is a real design goal here, not marketing, but it's an optimizer outcome you can usually rely on, not a language-level guarantee that holds unconditionally in every build.

Field study first

A service like this might validate every registered check for a controller before it starts accepting transfer requests: 

checks.iter().try_for_each(OperationalCheck::validate)?;

OperationalCheck::validate — the exact trait you designed the shape of in Lesson 11, written as a plain function reference. try_for_each calls it on every item in the list, stopping at the first error and propagating it out via ?. One line replaces a manual loop with a match inside it.

The theory

  • Iterators are lazy. .iter() doesn't do anything by itself — it produces a sequence of borrowed items that only actually run when something consumes them (.collect(), .for_each(), a for loop, etc.).
  • Closures (|x| x + 1) are anonymous functions you can pass around — Python's lambda, with the same idea: capture some context, take arguments, return a value.
  • A method reference like OperationalCheck::validate can be passed anywhere a closure of the matching signature is expected — no |x| x.validate() needed, since that's exactly what the method reference already means.
  • try_for_each is for_each's fallible cousin: it stops early and returns the first error instead of running to completion regardless.

Why this matters for the backend

A real backend's readiness checks, iteration over access runs, and much of its audit-log processing lean on exactly this style. Recognizing .iter().try_for_each(SomeTrait::method) as "loop over this, call this trait method, stop on first error" turns a dense one-liner into something you can read at a glance.
