Lesson 18: Cargo Workspaces

One level up from Lesson 17's module split: splitting one crate into many, and knowing exactly which ones a given Cargo command actually touches.

Why this matters

Lesson 17 split one crate's code into files. A real backend takes the same underlying pressure — "this grew too large to be one thing" — one level further: it's not one crate with many modules, it's several separate crates in one workspace (per the mission glossary: access-service, telemetry-core, test-support, and more as a service grows). Knowing what a workspace actually is, and which command builds which subset of it, is the difference between cargo build quietly doing far less than you assumed and knowing exactly what just happened.

Field study first

A backend workspace's root Cargo.toml might look like this (often preceded by a [workspace.package] section with shared metadata, so the [workspace] table itself sits a little further down than line one): 

[workspace]
default-members = ["crates/access-service"]
members = ["crates/*"]
resolver = "3"

One workspace, several member crates (access-service, telemetry-core, test-support, and others as the service grows) sharing one Cargo.lock and one target/ build directory. Notice: there's no [package] section anywhere in this file — this root Cargo.toml only ever defines the workspace itself, never a crate of its own (a virtual workspace, in Cargo's terms).

The theory

  • A workspace groups multiple actual crates (each with their own Cargo.toml) so they share dependency resolution and a build cache, without being one monolithic crate. A backend like this uses that to keep access-service, telemetry-core, etc. as independently buildable, independently testable units.
  • default-members is not "all members." When you run a package-selecting command (build, check, test, clippy) from the workspace root with neither -p/--package nor --workspace, Cargo operates on default-members only — in the field study above that's just crates/access-service. A bare cargo build at that workspace's root does not build every crate; it builds one.
  • --workspace forces the command onto every member, ignoring default-members entirely. This is what actually builds/checks/tests every crate in the workspace at once.
  • -p <package> (short for --package) targets exactly the named crate, regardless of default-members — and works from anywhere inside the workspace, not just the root.

Today's task

  1. Vim reminder — before you start this one gg / G — jump to the first/last line of a file you're skimming, not editing

    This task is mostly reading small Cargo.toml files and comparing command output, not editing Rust. gg/G (top/bottom of file) are the two jumps you'll actually want, moving fast between a crate's [package] name and the root's [workspace] table without scrolling by hand.

    Predict, then verify, what three build commands each touch

    Build a miniature two-crate workspace that mirrors the field study's own shape — small enough for immediate, offline feedback, no external registry access needed. From your projects directory (outside rust-practice, which is already a single crate, not a workspace): 

    mkdir -p workspace-demo/crates/access-service/src
mkdir -p workspace-demo/crates/telemetry-core/src
cd workspace-demo

    Root Cargo.toml — the same three keys as the field study's, on purpose:

    [workspace]
members = ["crates/*"]
default-members = ["crates/access-service"]
resolver = "3"

    crates/access-service/Cargo.toml:

    [package]
name = "access-service"
version = "0.1.0"
edition = "2024"

    crates/access-service/src/lib.rs: pub fn ping() -> &'static str { "access-service" }. Then the same shape for telemetry-core — crates/telemetry-core/Cargo.toml with name = "telemetry-core", and crates/telemetry-core/src/lib.rs returning "telemetry-core" instead.

    Before running each command below, write down which crate(s) you think it will compile. Then run it and check ls target/debug/*.d — one .d file per crate actually compiled, immediate ground truth: 

    cargo build
ls target/debug/*.d

cargo build --workspace
ls target/debug/*.d

cargo build -p telemetry-core
ls target/debug/*.d

    The first should show only access-service's output; the second adds telemetry-core; the third works even freshly (try it after rm -rf target) and touches only telemetry-core, never consulting default-members at all.

Tangible win Three commands, three different, correctly-predicted sets of compiled crates — not a claim you took on faith, but one you watched target/debug/ confirm each time.

Why this matters for the backend

A plain cargo build from a real backend workspace's root typically only builds its default crate — the same default-members behavior your demo just showed, at real scale. That's deliberate: most local iteration only needs the one crate you're actually changing. Exercising the full workspace is what cargo build --workspace (and whatever a given team's CI pipeline runs) is actually for. If you already have access to a larger multi-crate workspace, try cargo build -p <some-crate> there and confirm it compiles just that one crate — but this step is optional and not required to finish this lesson.

2. src/foo.rs, or src/foo/mod.rs for the directory form (Lesson 17).

Ask your tutor

If default-members vs. --workspace vs. -p still feels like three ways to say the same thing after running all three, ask — the distinction is small in syntax but real in consequence (compiling 1 crate vs. many is a very different wait), and it's worth being certain rather than guessing which one CI or a teammate meant.
