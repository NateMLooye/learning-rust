Lesson 15: Conflicts & jj undo, For Real

jj treats conflicts as a normal, first-class state — not an emergency. And jj undo is always safe, not just something to read about.

Why this matters

In git, a conflict stops you mid-operation and demands resolution before you can do anything else. In jj, a conflict is just... a state a change can be in. You can keep working, look at other changes, even commit on top of a conflicted change, and resolve it whenever you're ready. This is a real, deliberate design difference — worth experiencing once under low stakes.

Today's tasks (in rust-practice)

  1. Manufacture a real conflict, from a common base

    A real conflict comes from two sibling changes that both edit the same line, not from a change and something built on top of it — rebasing an ancestor onto its own descendant isn't a meaningful operation, there's no "before" and "after" there, just one line of history. Build a small, disposable fork instead, in a dedicated file so nothing real is at risk: 

    jj new -m "conflict-demo: base"
echo "original line" > access-policy.txt

    Run jj log and note this change's ID — call it <base> for the rest of this exercise. Now create a left child of it: 

    jj new -m "conflict-demo: left"
echo "left version" > access-policy.txt

    Note this change's ID too — call it <left>. Now go back and create a right sibling — a second child of <base>, not of <left> — by naming <base> explicitly. Substitute the real ID you noted above wherever you see <base>: 

    jj new -m "conflict-demo: right" <base>
echo "right version" > access-policy.txt

    jj log should now show <base> with two children — left and right — each disagreeing about the same line, neither one a descendant of the other. Rebase right (your current change, @) onto left and watch what happens — substitute your own <left> ID: 

    jj rebase -r @ -d <left>

    This replays the right edit on top of left instead of on top of the shared base — and since both sides touched the same line differently, jj can't merge them silently.

  2. Look at the conflict without panicking

    jj st will tell you access-policy.txt is conflicted. Open the file. jj does not render git's plain three-marker format here — it materializes a richer conflict, showing one side as a diff against the shared base and the other as plain content, roughly like this (your own change IDs and hashes will differ): 

    <<<<<<< Conflict 1 of 1
%%%%%%% diff from: <base> ... "conflict-demo: base" (parents of rebased revision)
       to: <left> ... "conflict-demo: left" (rebase destination)
-original line
+left version
+++++++ <right> ... "conflict-demo: right" (rebased revision)
right version
>>>>>>> Conflict 1 of 1 ends

    Nothing is blocking you. You could switch to a different change entirely, or run jj diff to see the same conflict summarized from the outside instead of materialized into the file. cargo test in the practice crate is unaffected either way — access-policy.txt isn't part of the Rust build.

  3. Vim reminder — before you start this one /<<<<<<< — search forward to the first conflict marker

    /<<<<<<< — a plain text search, not a mnemonic command — jumps forward to the first conflict marker, faster than scrolling to find where a conflict starts in a longer file. n repeats the same search forward if there's more than one conflicted region.

    Resolve it

    Edit access-policy.txt directly, remove every marker line (<<<<<<< through >>>>>>> and the %%%%%%%/+++++++ lines in between), keep whichever version (or a hand-merge of both) makes sense, save. Run jj st again to confirm the conflict is gone.

  4. Now break something on purpose, and recover

    Run jj abandon <base>:: (the :: means "this change and everything descended from it," so this removes the whole conflict-demo fork — base, left, and right — in one operation). Then: 

    jj undo

    Confirm with jj log that all three conflict-demo changes are back. This is the actual safety net, exercised on purpose instead of discovered by accident under pressure — and it works exactly the same way on a multi-change abandon as it did on the exercise abandon above.

Going further back

jj undo only reverts the single most recent operation. For anything further back, jj op log shows the full operation history, and jj op restore <op-id> restores the repo to any prior state — not just one step back.

Clean up when you're done

access-policy.txt and its three changes were disposable, purely to manufacture a conflict safely — once you're satisfied you've seen it, resolved it, and recovered from a mistake, remove them from your real history with the same command from the step above, this time for real (no jj undo after): 

jj abandon <base>::

Confirm with jj log that the three conflict-demo changes are gone and jj st shows access-policy.txt no longer tracked.

Tangible win A conflict you caused, saw, and resolved without anything breaking — and a genuine mistake you recovered from with one command.

Why this matters for the backend

Real conflicts happen constantly whenever more than one person touches the same files in a real backend codebase. The panic response ("did I just lose my work?") is the wrong instinct in jj — this lesson exists so that instinct isn't your first experience of a real conflict on real work.

Primary source

The Jujutsu tutorial, section on conflicts as first-class state. No Comprehensive Rust or Rustlings equivalent — neither teaches a VCS.

Ask your tutor

If the conflict markers look unfamiliar or the rebase command's flags (-r/-d) are confusing, ask before resolving — better to understand what you're editing than to guess and get lucky.
