Lesson 12: jj Squash & Split

Reshaping messy history into clean changes, after the fact instead of trying to get it perfect the first time.

Why this matters

Real work is never as clean as "one lesson, one change." You'll fix a typo in something you already described, or describe one change that actually contains two unrelated things. Squash and split are how you clean that up after the fact, instead of trying to get it perfect the first time.

Today's tasks (in rust-practice)

  1. Vim reminder — before you start this one cw — change from the cursor to the end of the word

    Same change operator as ci{/ci", different text object: cw is change + word — it changes from the cursor to the end of the current word, dropping you into insert mode. Exactly the right tool for "a small doc-comment wording issue" below.

    Make a change that should be two

    In one sitting, without describing in between, make two unrelated, real edits in the same file — this is about jj, not new Rust, so both edits use code you already have: fix a small doc-comment wording issue on an old function (e.g. tighten ready_count's doc comment from Lesson 5), and add one new #[test] covering an edge case an existing function doesn't yet test (e.g. an AccessRun with an empty outcomes list — ready_count should return 0). Run cargo test once to confirm the new test passes before moving on, then run: 

    jj split

    With no path given, jj split opens an interactive diff editor listing every changed hunk in the file. Select only the doc-comment fix and leave the new test unselected, then confirm. The hunks you selected become a new change; whatever you left unselected stays behind in the change you started from. Run jj log to see the two changes it produced, then jj diff -r <change-id> on each one to confirm the doc-comment fix and the new test landed in different changes.

  2. Make a change that should be one

    Describe a change, then realize you forgot something small and directly related (a missed test case, a typo in the same function). Instead of describing a second unrelated-looking change: 

    jj new
# make the small follow-up edit
jj squash

    jj squash folds the current change into its parent, keeping the parent's description. Run jj log and confirm the follow-up edit is no longer a separate entry.

Recovery, always

Both operations are ordinary jj operations — jj undo reverts either one cleanly if the split or squash didn't go the way you expected.

Tangible win You reshaped messy history into clean changes twice — once by splitting a mixed-together edit apart, once by folding a small follow-up into its rightful parent.

Why this matters for the backend

Review feedback on a real backend-service change almost always wants clean, logically separated commits — "this is the fix, this is the test, these are unrelated and should be their own change." Squash and split are how you get there without redoing work from scratch.

Primary source

The Jujutsu tutorial, sections on jj squash and jj split. No Comprehensive Rust or Rustlings equivalent — neither teaches a VCS.

Ask your tutor

The interactive diff editor for jj split (no paths given) has its own learning curve — if the keybindings or UI aren't obvious, ask rather than guessing your way through an unfamiliar TUI under time pressure.
