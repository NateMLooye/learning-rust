Lesson 17: Modules & Visibility

Your lib.rs has a dozen lessons crammed into one file by now. Time to split it up — the same mechanism a real backend crate uses to organize itself into files.

Why this matters

By now src/lib.rs has functions, structs, enums, and traits from a dozen different lessons all in one file. That's exactly the pressure that makes real codebases split into modules — a smaller-scale version of the same pressure that eventually pushes a crate to split into a multi-crate workspace, which is its own lesson right after this one.

Field study first

A backend crate for this domain might declare its modules like this: 

pub mod access;
/// Telemetry ingestion from vessel and asset sensor feeds
pub mod telemetry;
/// Access-window and operating-parameter policy evaluation
pub mod policy;
pub mod health;
pub mod concurrency;

Five pub mod declarations, each one covering ground you've effectively already read about in earlier field studies — access mirrors Lesson 4's AccessOutcome modeling, policy mirrors Lesson 9's operating-parameter merging, concurrency mirrors Lesson 16's Arc-shared ControllerState. health is the one exception: there's no health.rs, because it's a directory with its own src/health/mod.rs — the second form the theory below describes.

The theory

  • mod declares a module. mod foo; in lib.rs tells Rust to look for the module's code in src/foo.rs (or src/foo/mod.rs — the form health uses above). Everything inside is private by default; pub mod foo; or pub fn/pub struct inside it makes specific things visible from outside.
  • A module is a namespace, not a new crate. Splitting a file into modules doesn't change what the code does — it changes how it's organized and what's visible to what. This is a much smaller step than splitting one crate into many, but the same underlying motivation: one giant file becomes hard to navigate.
  • pub use re-exports a name through a new path. pub use basics::add; in lib.rs means callers can still write rust_practice::add even after add moves into basics.rs — the public API stays flat while the file layout underneath it changes.

Today's tasks

  1. Warm up with a Rustlings rep

    Before splitting lib.rs below, practice module visibility and paths on their own, on a small file where getting pub wrong doesn't put a dozen lessons' code at risk. Copy just the first file of Rustlings' 10_modules set, modules1.rs, into rust-practice/src/bin/ and get it green — see the tooling cheatsheet's "Wiring a Rustlings exercise into rust-practice" for the exact steps (note: this warm-up still lives under src/bin/ as its own binary target — a separate mechanism from the mod split you're about to do inside the library target itself). For more depth than the theory section above covers, the Rust Book's chapter 7, "Managing Growing Projects with Packages, Crates, and Modules" goes further.

    Optional extra practice: the rest of 10_modules (modules2.rs onward) is good additional reps, not required for this lesson.

  2. Vim reminder — before you start this one yap / dap — yank a function into a new file, then delete the original

    This refactor is genuinely about moving text between files, not editing in place. In the old lib.rs buffer, yap (yank + around + paragraph — the same pattern from Lesson 14, blank lines act as paragraph boundaries in Rust) yanks a whole function; open the new file with :e src/access.rs, p (paste) to drop it in, then delete the original with dap (delete + around + paragraph) back in lib.rs once it's safely moved.

    Split your lib.rs into modules — a real refactor, no failing test to guide you

    This time the "test" is that your existing tests still pass after the refactor — that's the actual skill: trusting your test suite through a structural change instead of a new feature.

    Pick a grouping that makes sense to you. One reasonable split, using the same four module names as the field study above:

    src/
  lib.rs          -- just `mod` declarations + `pub use` re-exports
  access.rs       -- AccessOutcome, AccessRun, summarize, ready_count, Outcome trait, count_ready (Lessons 4, 5, 11, 14)
  telemetry.rs    -- resolve_access_label, both_versions, describe_parameter, ParameterName (Lessons 6, 7, 13)
  policy.rs       -- merge_operating_parameters, parse_heave_limit, lookup_heave_limit_override (Lessons 9-10)
  concurrency.rs  -- AccessEvent, record_access_event (Lesson 16)

    Lessons 1-2's tiny warm-ups (add, select_conservative_limit) don't fit any of the four cleanly — leave them in lib.rs directly, or fold them into access.rs, your call either way.

    In lib.rs, replace the flat contents with module declarations: mod access;, mod telemetry;, and so on. Move each group's functions/enums/traits into its own file. Your tests so far are plain #[test] fns sitting directly next to the function they test — not yet wrapped in the convention many real Rust codebases use, a #[cfg(test)] mod tests { ... } block at the bottom of the file. Moving code into files is a natural moment to adopt that convention too: gather each file's #[test] fns into one mod tests block at the bottom, with use super::*; at the top so they can still see the functions above without repeating the module path.

    Run:

    cargo test

    Every test from every previous lesson should still pass, unchanged. If something doesn't compile, it's almost always a missing pub or a wrong use path — both fixable with rust-analyzer's diagnostics from Lesson 2.

Tangible win A dozen lessons' worth of code, reorganized into real modules, with every single test still green — proof the refactor changed nothing about behavior, only structure.

Why this matters for the backend

You just did, at file scale, what a real backend also does at crate scale: split one thing that grew too large into named, purpose-specific pieces, without changing what any of it does. The next lesson takes this same idea up one level — a real workspace's multi-crate layout.

Worth a clean change

This is a pure structural refactor with no behavior change — a good candidate for its own cleanly described jj change, separate from whatever feature work comes next. jj new before starting, jj describe when the tests are green.

Ask your tutor

If you're unsure whether a function should be pub or stay private after the split, ask — visibility choices in module design are a real decision, not a formality, and worth reasoning through rather than just adding pub everywhere until it compiles.
