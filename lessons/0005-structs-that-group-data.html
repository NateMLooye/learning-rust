Lesson 5: Structs That Group Data

Rust's plain data-grouping tool — no inheritance, no hidden behavior, just named fields and the functions you choose to attach to them.

Why this matters

Lesson 4's AccessOutcome models "one of several possibilities." Most real data isn't a choice between variants — it's several pieces of information that belong together at once. A vessel visit has a vessel name and a site and an access window, all simultaneously. That's what a struct is for, and it's the single most common shape you'll see across every backend file you open from here on.

Field study first

A service like this might model visits and requests with structs like these, all within a similarly small span of one file: 

pub struct VesselVisit {
    pub vessel: String,
    pub site: AssetId,
    pub window: AccessWindow,
}

pub struct TransferRequest {
    pub id: u64,
    pub personnel: u32,
}

pub struct AccessWindow {
    pub opens_at: u64,
    pub closes_at: u64,
}

Notice the shape repeats: a handful of named fields, most of them plain values or other domain types, all marked pub. A whole visit record is, structurally, nothing more than a VesselVisit holding an AssetId and an AccessWindow — structs nested inside structs, all the way down.

Now a struct paired with a real impl block: 

#[derive(Clone)]
pub struct CompensationReading {
    pub heave_m: f64,
    pub ok: bool,
}

impl CompensationReading {
    /// Returns true if the reading is within the safe heave envelope.
    pub fn is_safe(&self, max_heave_m: f64) -> bool {
        self.ok && self.heave_m.abs() <= max_heave_m
    }
}

Two fields, one struct, one method. impl CompensationReading { ... } attaches functions to the type from outside its own definition — is_safe takes &self to look at the reading without consuming it, plus one extra parameter, and returns a fresh bool.

The theory

  • A struct groups named fields into one type. struct Name { field: Type, ... } defines it; Name { field: value, ... } creates ("instantiates") one. Every field must be given a value at construction — there's no partial construction, no implicit null/None default, unless you write one yourself.
  • Field access is direct. visit.vessel reads the field, plain and simple — no getter method required, no boilerplate. VesselVisit, TransferRequest, and AccessWindow all mark their fields pub and leave it at that; a getter method that just returns a field verbatim would add a layer of indirection without hiding anything.
  • impl Name { ... } attaches functions to the type. A function whose first parameter is &self is a method, called with instance.method() — same borrowing rules from Lesson 4's &AccessOutcome apply here, &self just means "look, don't consume." A function with no self parameter is an associated function, called with Type::function(...) instead — String::from(...), which you've already used, is exactly this.
  • No inheritance between structs. Unlike a Python class hierarchy, there's no "struct extends struct." Shared behavior across different structs comes from traits (Lesson 11), not from one struct being a subtype of another.
  • One small piece of syntax for the exercise below: every Vec<T> has a .len() method that returns how many elements it holds — the equivalent of Python's len(some_list), just spelled as a method instead of a built-in function. Its return type is usize: an unsigned (never negative) integer sized to match the platform's pointer width, Rust's standard type for lengths, counts, and indices. Lesson 9 covers Vec properly; this is just enough to call one method on it.

Why this matters for the backend

"A handful of named fields, an impl block attaching behavior" is the single most repeated shape in a real backend codebase — VesselVisit, TransferRequest, AccessWindow, CompensationReading, and dozens more all follow it. Once you can recognize a struct definition and its impl block on sight, a new backend file stops looking unfamiliar and starts looking like a variation on a pattern you already know.
