Lesson 16: Shared State & Arc

Lesson 6 taught single ownership. This is what Rust does when a value genuinely needs more than one owner at once.

Why this matters

Lesson 6 established: one owner per value. But a real backend's shared controller state needs to be accessible from many handlers at once while the service runs — that's not a bug in the ownership model, it's a real requirement Arc exists to satisfy.

Field study first

A service like this might wrap Lesson 9's ControllerState in exactly this shape: 

pub type Ctx = Arc<ControllerState>;

pub struct ControllerState {
    pub active_visits: HashMap<AssetId, VesselVisit>,
    pub parameters: HashMap<String, String>,
    pub emergency_stop: AtomicBool,
    log: Mutex<Vec<AccessEvent>>,
}

Ctx — the type a handler would take to reach the running service's shared state — is literally Arc<ControllerState>. Every part of a running service holds one of these, and they all point at the same underlying ControllerState: the same active_visits and parameters fields from Lesson 9, plus an emergency_stop flag and an event log that specifically need to be mutated safely from more than one place at once.

The theory

  • Arc<T> ("Atomically Reference Counted") gives you shared ownership: cloning an Arc doesn't copy the inner data, it just increments a counter and hands back a pointer to the same value. When the last Arc pointing at it is dropped, the value is freed. This is how multiple owners coexist without violating Lesson 6's single-owner rule for the underlying data — the Arc wrapper is what's actually cloned, not the thing inside.
  • Arc alone only gives you shared read access. To mutate shared data safely across threads, you need something inside the Arc that manages concurrent access itself — the standard-library answer is Mutex<T> (lock, mutate, unlock), which is exactly what guards ControllerState.log above.
  • AtomicBool is a lighter-weight tool for a single boolean — safe concurrent read/write with no lock at all, using CPU-level atomic instructions. An emergency-stop flag is a natural fit: many places need to check or set one bit of state, cheaply, without contending over a full lock.

    Shared mutable state, the std-library way

    Add this scaffold (no new dependency needed — Arc and Mutex are both in std):

    use std::sync::{Arc, Mutex};

/// A single audit-log entry.
#[derive(Debug, Clone, PartialEq)]
struct AccessEvent {
    asset: String,
    outcome: String,
}

/// Appends `event` to the shared log and returns its new length. Takes
/// ownership of an `Arc` handle rather than borrowing one, the same way a
/// real handler takes ownership of its own `Ctx` clone.
fn record_access_event(log: Arc<Mutex<Vec<AccessEvent>>>, event: AccessEvent) -> usize {
    todo!()
}

#[test]
fn record_access_event_appends_and_returns_length() {
    let log = Arc::new(Mutex::new(Vec::new()));
    let arrival = AccessEvent { asset: "gangway-04".to_string(), outcome: "ready".to_string() };
    let blocked = AccessEvent { asset: "gangway-04".to_string(), outcome: "blocked".to_string() };
    let departure = AccessEvent { asset: "gangway-04".to_string(), outcome: "ready".to_string() };

    assert_eq!(record_access_event(Arc::clone(&log), arrival), 1);
    assert_eq!(record_access_event(Arc::clone(&log), blocked), 2);
    assert_eq!(record_access_event(Arc::clone(&log), departure), 3);

    // Read through the original `log` handle — a fourth owner of the same
    // underlying Vec, never passed into `record_access_event` at all.
    assert_eq!(log.lock().unwrap().len(), 3);
}

    Write the body yourself. You'll need .lock() (which returns a Result — .unwrap() is fine here) to get access to the Vec inside, then push and read its length. Notice the test can't compile with log passed directly (moved) into record_access_event more than once — record_access_event takes ownership of its Arc parameter, so calling it three times with the same underlying log requires Arc::clone(&log) at each of those three call sites; that's the multiple-ownership idea from the theory section, forced by the compiler rather than just asserted. Get cargo test record_access_event_appends_and_returns_length green.

Optional: count the handles

Add assert_eq!(Arc::strong_count(&log), 1); right before the three record_access_event calls in the test above. Because each Arc::clone(&log) handle you pass into record_access_event is dropped as soon as that call returns, the strong count only ever reflects handles alive right now — a concrete look at what "reference counted" actually counts, moment to moment, rather than a running total.

Tangible win Shared, safely-mutated state across four distinct Arc handles pointing at the same underlying data — the std-library equivalent of exactly what Ctx = Arc<ControllerState> does for real, at scale, in a production backend.

Why this matters for the backend

Every handler receiving a Ctx is receiving a clone of the same Arc<ControllerState> — cheap to pass around, all pointing at the one real shared state for that running service. You now know exactly what that type alias is doing and why it's not just "a smart pointer for no reason."

Answer from memory before opening the reveal:

  1. What does a trait bound like T: ToString actually unlock inside a generic function that plain T doesn't?
  2. How does jj treat a conflict differently from how git treats one?
Reveal the answers

1. Without a bound, the compiler only allows operations every possible type supports — almost nothing. T: ToString unlocks .to_string() and anything else the bound's trait provides (Lesson 13).

2. jj treats a conflict as ordinary, first-class state — you can keep working, switch changes, even build on top of a conflicted change, and resolve it whenever you're ready. git stops you mid-operation and demands resolution first (Lesson 15).
