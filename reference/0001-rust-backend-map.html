Reference: Rust ↔ Backend Concept Map

Every Rust concept in this course, paired with where the equivalent shape shows up in a backend service.

This is the compressed version of the field study: for each Rust concept you'll practice, here is exactly where it shows up in the course's synthetic access-backend domain. Read the referenced file before the concept feels solid, to see it used for real, and come back to this table any time you need to re-locate an example.

The synthetic domain

This course's examples come from a small, made-up Cargo workspace of three crates: access-service (evaluating and recording transfer requests), telemetry-core (aggregating reported status), and test-support (shared test fixtures). Every type named below — AccessOutcome, AccessRun, ControllerState, and so on — is a synthetic teaching construct defined in MISSION.md's domain glossary, not real safety or motion-control code. It's a coherent, deliberately simple corpus: enums, traits, generics, Result/anyhow, and shared state, without the scarier corners of the language getting in the way.

Concept map

Rust concept Where it shows up (backend) What to notice
Enums + pattern matching access-service/src/outcome.rs (AccessOutcome: Ready, Blocked(reason), EscortRequired) Enums are the central modeling tool here, closer to a Rust superpower than a curiosity. Note #[non_exhaustive]-style thinking and how a data-carrying variant like Blocked(String) keeps the reason attached to the outcome.
Structs, impl, Display/Debug access-service/src/run.rs (AccessRun)
telemetry-core/src/status.rs (telemetry structs)		 Small, self-contained files. A struct, its impl block, and a couple of standard trait implementations — the shape you'll write a hundred times.
Ownership & borrowing access-service/src/run.rs (the evaluate/audit method on AccessRun) Watch where a reference is borrowed vs. where .clone() is reached for. This is where "who owns this?" stops being abstract.
Lifetimes (elided & explicit) access-service/src/labels.rs (resolve_access_label<'a>, a borrowed-label resolver) Most references need no annotation at all — elision handles the common case. A function that borrows from two different inputs and returns one of them is the case elision can't handle: the compiler needs the lifetime spelled out.
Result, ?, anyhow vs. thiserror access-service/src/outcome.rs (thiserror)
access-service/src/config.rs (parse_heave_limit, anyhow, ?, .context())		 Two error idioms coexist on purpose: thiserror typed enums for library code, anyhow + bail!/.context() for application code such as parsing an operating parameter. Notice which file uses which and why.
Traits (incl. From/TryFrom) access-service/src/outcome.rs (the Outcome trait)
access-service/src/checks.rs (the OperationalCheck trait)		 A one-method trait implemented across a set of checks via try_for_each(OperationalCheck::validate) — a clean, minimal example of Rust's interface mechanism.
Generics & trait bounds access-service/src/config.rs (describe_parameter<T: ParameterName>) Generic functions with a trait bound in the signature — the Rust equivalent of Python's duck typing, but checked at compile time.
Iterators & closures access-service/src/checks.rs (try_for_each(OperationalCheck::validate)) Iterator combinators over a list of checks, passing a trait method as a closure-like argument.
Arc, Mutex, shared state telemetry-core/src/state.rs (type Ctx = Arc<ControllerState>, Mutex, AtomicBool) Concurrent shared state across handlers — a good bridge from "ownership is single-threaded" to "here's how Rust shares safely across threads."
Modules & Cargo workspace root Cargo.toml (members = ["access-service", "telemetry-core", "test-support"]) A small workspace of three crates. This is the Rust equivalent of a Python monorepo with per-package pyproject.toml files.
Testing idioms inline #[cfg(test)] mod tests throughout, e.g. access-service/src/outcome.rs Tests live inline in the file they test. No separate test framework to learn — cargo test is the whole story.
Advanced: async trait objects, a parser read and admire, not build-your-own Real backend services often have at least one file storing async closures as trait objects, or a hand-written parser for some config grammar. Worth reading once you find one, but not a "build one yourself" target in this course.

Suggested reading order

Roughly easy → hard, if you want to browse ahead of the lessons:

  1. test-support/src/lib.rs — small, the shared fixtures crate.
  2. access-service/src/outcome.rs — a clean, self-contained thiserror enum plus the Outcome trait.
  3. access-service/src/checks.rs — one small trait (OperationalCheck), end to end.
  4. access-service/src/config.rs — parse_heave_limit with let-else, ?, .context().
  5. access-service/src/run.rs — AccessRun, the evaluate/audit method.
  6. access-service/src/labels.rs — resolve_access_label, explicit lifetimes.
  7. telemetry-core/src/state.rs — Arc + Mutex + AtomicBool shared-state ControllerState.
