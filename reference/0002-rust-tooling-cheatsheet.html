Reference: Rust Tooling Cheatsheet

Cargo, rust-analyzer, an editor/LSP, and version-control commands — one page.

This is the command reference to keep open in a terminal pane. It does not explain why each tool exists in depth — see RESOURCES.md for the primary sources — it's for quickly recalling the exact incantation.

Cargo

CommandWhat it does
cargo new --lib <name>Scaffold a new library crate (mirrors this course's inline-test style).
cargo checkType-check without producing a binary. Fastest feedback loop.
cargo buildCompile the crate/workspace (debug profile by default).
cargo runBuild and run the binary target.
cargo testRun every #[test]/#[tokio::test] in the crate.
cargo test <substring>Run only tests whose name contains the substring — e.g. cargo test lesson_01.
cargo clippyLint for idiomatic/correctness issues beyond what the compiler checks.
cargo clippy --workspaceSame, across every crate in a workspace.
cargo fmtAuto-format the code per rustfmt.toml.
cargo fmt --checkVerify formatting without changing files — what CI runs.

Wiring a Rustlings exercise into rust-practice

Several lessons point at a single Rustlings exercise set as a warm-up rep, done before that lesson's domain-tied task — the same mechanic, isolated, with no domain context to juggle yet (see the Rustlings cross-map for how each lesson's pick was chosen). The wiring is identical every time, so it's written once here instead of repeated in every lesson.

Each linked set is a small directory of a few numbered files, not one file — e.g. exercises/02_functions/ holds functions1.rs through functions5.rs plus a README.md. A lesson's link goes to that directory so you can browse it; grab the actual .rs files with one of these:

  1. One-off, no local clone: curl each numbered file straight from raw.githubusercontent.com (swap in the set/file you want): 
    curl -s -o rust-practice/src/bin/functions1.rs \
  https://raw.githubusercontent.com/rust-lang/rustlings/main/exercises/02_functions/functions1.rs
    -o <path> renames on the way in; drop it and use -O instead to keep Rustlings' own filename. curl ships with macOS by default — wget doesn't (you'd need brew install wget), so curl is the native tool here, not a workaround.
  2. Doing this across several lessons? Clone once, reuse. Git's unit of cloning is a whole repository — there's no direct "clone one file." The practical equivalent, since the whole Rustlings repo is small, is a one-time shallow clone kept outside rust-practice: 
    git clone --depth 1 https://github.com/rust-lang/rustlings rustlings-src
    then just cp whichever numbered files a lesson needs, with no repeated network calls for the rest of the course: 
    cp rustlings-src/exercises/02_functions/functions1.rs rust-practice/src/bin/
  3. Either way, land each numbered file you're using directly under src/bin/ as its own file — they're independent (no shared state between functions1.rs and functions2.rs), so one file per binary is the natural fit. No Cargo.toml edit needed — Cargo auto-discovers every .rs file directly under src/bin/ as its own binary target, named after the file. See the Cargo Book's binary target reference for the mechanism.
  4. Run one: cargo test --bin functions1. Run everything you've wired up so far (every src/bin/*.rs plus lib.rs) at once with plain cargo test.
  5. Fix each file until it's green, then move on to the lesson's domain-tied task below it — that's the entire warm-up.
Ask your tutor

If two exercises would want the same filename under src/bin/ (e.g. two sets both happen to have a file that'd land as errors1.rs), rename one — the binary name just needs to be unique within the crate, not identical to Rustlings' own filename.

rust-analyzer & your editor

Whatever editor you use, install rust-analyzer and wire it up as an LSP client — most modern editors either bundle Rust support or have a one-line plugin/extension for it.

ThingWhat it gives you
rust-analyzer (LSP)Live diagnostics as you type, hover docs, go-to-definition/references, inline type hints, rename, code actions — the "glue" that talks to rustc/cargo/clippy for you.
editor LSP status commandMost editors expose some way to confirm rust-analyzer is actually attached to the current buffer (e.g. an "LSP info" command) — use it if diagnostics seem to have stopped updating.
editor health-check commandGeneral editor health check — useful when something seems misconfigured.

Treesitter specifically

If your editor uses tree-sitter-based syntax highlighting, make sure the rust (and ron, if you touch RON config files) parsers are installed. Parser compilation needs the tree-sitter CLI as a separate binary from cc/gcc — install it if a parser install fails with something like ENOENT ... 'tree-sitter'.

Known gotchas

A brand-new src/bin/<name>.rs can show "this file is not included anywhere in the module tree" right after you create it (exactly the warm-up recipe above does, every lesson) — rust-analyzer builds its crate graph from cargo metadata once at startup/last reload and doesn't always notice a new binary target appearing after that; its fallback guess for an unrecognized loose .rs file is "this must be a module of the nearest crate, reachable via mod," which is what that message is actually describing. It's a stale-index problem, not a real one: cargo test --bin <name> in a terminal already works regardless, since Cargo itself (not rust-analyzer's cache) is what scans src/bin/. Fix the warning by restarting the LSP client, or just close and reopen the editor if you don't have that command bound.

clippy-on-save and format-on-save: most editor LSP setups can run clippy instead of plain cargo check for diagnostics, and run rustfmt automatically on save — check your editor's LSP/formatter config if you want either behavior and it isn't already on.

jj (Jujutsu)

CommandWhat it does
jj stStatus — what's changed in the working copy (which is itself a commit in jj).
jj logShow recent changes/history graph.
jj diffShow the diff of the current change.
jj describe -m "..."Set/replace the description of the current change.
jj newStart a new (empty) change on top of the current one.
jj squashFold the current change into its parent.
jj splitInteractively split the current change into two.
jj undoUndo the last jj operation — always safe, the universal escape hatch.
jj op logFull operation history, for finding a state to restore to.
jj git push --bookmark <name>Push a bookmark (jj's version of a branch) to the git remote.
Recovery, always

If any jj history operation goes wrong — bad squash, wrong rebase, accidental abandon — reach for jj undo first, or jj op log + jj op restore <op-id> for anything further back. Never rm -rf .jj.

Project build commands

Real backend workspaces usually wrap the raw cargo commands above in a Makefile or a CI job target, so contributors run one short command instead of remembering flags. A typical wrapper might run cargo fmt --check, cargo clippy --workspace, and cargo test (or a faster test runner) in sequence — check your project's Makefile or CI config for the actual equivalent before assuming plain cargo test is everything CI does.
