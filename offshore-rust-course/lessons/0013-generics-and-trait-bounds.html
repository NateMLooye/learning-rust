Lesson 13: Generics & Trait Bounds

Compile-time duck typing — one function, many types, checked before it ever runs.

Why this matters

Python's duck typing means len(x) works on anything with a __len__, found out at call time. Rust generics get you the same flexibility — one function definition, many concrete types — but the compiler verifies every possible type satisfies the requirements before your code ever runs.

Field study first

Two synthetic one-liners, from a backend's telemetry module:

// access-service/src/telemetry.rs:41-43
pub fn text<T: ToString>(t: T) -> Self {
    Self::Text(t.to_string())
}

// access-service/src/lib.rs:118
pub fn new<I: Into<String>>(id: I) -> Self {

SensorReading::text accepts anything that implements ToString — an i64, a &str, a custom type, whatever — and calls .to_string() on it. SensorId::new accepts anything convertible into a String. Both patterns let callers pass a &str literal directly without an explicit .to_string() at every call site.

The theory

  • Generic syntax: fn name<T>(param: T) — T is a placeholder for "some type, decided by the caller." Without a bound, the compiler only lets you do things every possible type can do (which is almost nothing useful).
  • A trait bound (T: ToString) restricts T to types implementing that trait, unlocking its methods inside the function. This is the actual mechanism — generics and traits aren't separate topics, bounds are where they meet.
  • ToString and Into<String> are both real, ordinary traits — the same kind of thing you designed yourself in Lesson 11, just from the standard library instead of hand-written.

Optional: see the bound actually enforced

Define a tiny struct with no Display impl (just struct Thing;) and try calling describe_parameter(Thing). Read the compiler error — it names the exact trait that's missing. That's the "checked before it runs" guarantee from the top of this lesson, made concrete.

Tangible win Two generic signatures you designed end-to-end, using the same bound-on-a-standard-trait pattern the field study's backend code uses twice.

Why this matters for the backend

SensorReading::text and SensorId::new exist so callers throughout a real codebase can write SensorReading::text("42.1") instead of SensorReading::Text("42.1".to_string()) everywhere — a small ergonomic win that repeats often across a real backend codebase.

Ask your tutor

If T: ToString vs T: Into<String> vs T: std::fmt::Display feels like three names for the same idea, ask — they're related but genuinely different, and the difference explains why a real codebase might pick a different one for each of two similarly-shaped functions, the way the field study does.
