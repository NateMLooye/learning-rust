Lesson 11: Traits You Design Yourself

Rust's version of an interface — plus struct-shaped enum variants, a new bit of syntax hiding in plain sight in a backend service's own error type.

Why this matters

Python's duck typing works at runtime: you call .validate() on something and find out at call time whether it has that method. Rust traits check this at compile time — if a type doesn't implement the trait, your code doesn't build, full stop. Traits are how a backend service gets polymorphism without inheritance.

Field study first

A service like this might define a one-method trait for anything that can check its own readiness, plus an error enum covering what can go wrong: 

pub trait OperationalCheck {
    fn validate(&self) -> Result<(), String>;
}

pub enum ControllerError {
    NoActiveWindow,
    SiteUnreachable(String),
    InvalidSensor { channel: u8, reason: String },
}

Two things to notice:

  • OperationalCheck — a trait with exactly one method. Anything implementing it gets a uniform way to be validated, regardless of what it actually is.
  • ControllerError's variants are simple (NoActiveWindow, no data) or tuple-shaped (SiteUnreachable(String), like Lesson 4's Blocked(String)) — but InvalidSensor { channel, reason } is different: a struct-shaped variant, with named fields instead of positional ones. Same enum, three different variant shapes.

The theory

  • A trait defines a set of methods a type must implement to "count as" that trait — a contract, not a base class. There's no inheritance hierarchy. But it's not "any type, anywhere, can implement any trait," either: Rust's orphan rule (part of what it calls coherence) requires that either the trait or the type you're implementing it for be defined in your own crate. You can implement your own trait for String, or a std trait for your own type, but not Display (std) for Vec<T> (std) — that combination belongs to neither crate you own.
  • Defining your own trait: trait Name { fn method(&self) -> ReturnType; }. Implementing it for a type: impl Name for Type { fn method(&self) -> ReturnType { ... } }.
  • Struct-shaped enum variants (Variant { field: Type, ... }) work exactly like a small anonymous struct attached to that one variant — you pattern-match them with Variant { field, .. } the same way you'd match tuple variants, just by name instead of position.

    Design and implement a trait — no scaffold this time

    This continues the deliberate scaffold fading from Lessons 2 and 7: you have a description and a test only, no trait definition, no impl block. The failing test still exists before you design against it — it's just the whole interface, not one function's parameters/return, that's now yours to design:

    Define a trait Outcome with one method, passed(&self) -> bool. Implement it for AccessOutcome from Lesson 4. 
    #[test]
fn outcome_trait_on_access_outcome() {
    assert!(AccessOutcome::Ready.passed());
    assert!(!AccessOutcome::EscortRequired.passed());
    assert!(!AccessOutcome::Blocked("high seas".to_string()).passed());
}

    Write the trait definition and the impl block yourself. You'll need a match inside passed, the same mechanic from Lesson 4 — just returning a bool instead of a String this time. Get cargo test outcome_trait_on_access_outcome green.

Optional: a struct-shaped variant of your own

Add a fourth AccessOutcome variant, Faulted { code: u32, message: String } — struct-shaped, like the InvalidSensor variant on ControllerError from the field study above. Update summarize from Lesson 4 and your new passed impl to handle it (false, obviously). This forces the compiler's exhaustiveness check from Lesson 4 to catch every place you forgot to update — a real preview of what changing a shared type costs in a real backend codebase.

Tangible win A trait you designed from a one-sentence description, not copied or scaffolded — the actual interface-design skill, not just filling in a body.

Why this matters for the backend

A one-method trait like OperationalCheck implemented across many different types — sensors, controllers, gangway units — gives a backend service one uniform way to check readiness across all of them, through this one trait. Lesson 14 will show you the exact line that drives this across a whole list at once.
