Lesson 9: Collections & Strings

Vec, String/&str, and HashMap — the three data structures you've already been reading past in every field study so far, named properly at last.

Why this matters

Every lesson so far has used a Vec<...> field or a &str return type without stopping to explain it — Lesson 5's AccessRun.outcomes: Vec<AccessOutcome>, Lesson 7's label(&self) -> &str. Python's list and dict have direct Rust equivalents (Vec and HashMap), but Rust's are statically typed and heap-allocated on purpose, and Rust splits "a string" into two genuinely different types where Python has one. This lesson names all three properly, right before Lesson 10 needs you comfortable with a method that returns Option.

Field study first

A service like this might use two HashMaps in its controller module, playing different roles: 

// a function parameter
pub fn evaluate_transfer(
    &mut self,
    site: AssetId,
    request: TransferRequest,
    overrides: HashMap<String, String>,
) -> AccessOutcome {

// a struct field
#[derive(Debug, Default)]
pub struct ControllerState {
    pub active_visits: HashMap<AssetId, VesselVisit>,
    pub parameters: HashMap<String, String>,
}

overrides is a map from String to String — a caller's way of overriding specific operating parameters for one call without touching the global config. ControllerState.active_visits is different: AssetId keys to VesselVisit values, tracking which vessel is currently at which site. Same HashMap<K, V> type, two different key/value shapes.

Now a HashMap nested inside another collection — an audit log recording every event for every asset: 

#[derive(Debug, Clone, Default)]
pub struct AuditLog {
    // Maps each asset to every access event recorded for it.
    events: HashMap<AssetId, Vec<AccessEvent>>,
}

One asset can have several events (arrivals, departures, blocked requests) — so the value isn't a single AccessEvent, it's a Vec of them. This "one key, many values" shape is common enough to recognize on sight: HashMap<K, Vec<V>> reads as "a map where each key can have multiple values," not a mistake or an odd choice.

The theory

  • Vec<T> is a growable, heap-allocated list of one type — you've already used it as a struct field (Lesson 5's AccessRun.outcomes) without naming it. vec![a, b, c] builds one directly; Vec::new() starts empty; .push(item) appends.
  • String vs. &str. String is owned, growable, heap-allocated text — you construct one, you can mutate it (.push_str(...)), and it has exactly one owner (Lesson 6). &str is a borrowed view into string data — either into an existing String, or into text baked directly into the compiled binary (a string literal like "hello" is always &'static str). Lesson 7's label(&self) -> &str returns a borrowed view into self's own data — no new allocation, no ownership transferred, just a reference with a lifetime tied to self.
  • HashMap<K, V> associates keys with values: .insert(k, v) adds or overwrites an entry, .get(&k) looks one up and returns Option<&V> — Some(&value) if the key exists, None if it doesn't. Lesson 10 covers Option properly; for now, just know that "might not be there" is baked into the return type rather than raising an error or returning a sentinel value the way a Python dict[key] would raise KeyError.
  • Nested collections compose the same way structs do. HashMap<String, Vec<T>> is exactly what it looks like: a HashMap whose values happen to be Vecs, same as AuditLog.events above. There's no special syntax for nesting — it's the same generic type parameters you've already seen in Lesson 5's struct fields, just two levels deep.
  • Two small pieces of syntax for the exercise below. First, let mut x = ...; — every binding is immutable by default (Rust's answer to "did I mean to change this?"); adding mut is how you opt in to reassigning or mutating it later, e.g. x.insert(...) on a variable you plan to build up. Second, for (k, v) in some_map { ... } — a HashMap is directly iterable as key-value pairs, and this for form destructures each pair into k and v right in the loop header, the same way Python's for k, v in some_dict.items(): does.

#[test]
fn merge_operating_parameters_overrides_win() {
    let mut base = HashMap::new();
    base.insert("host".to_string(), "prod.example.com".to_string());
    base.insert("port".to_string(), "443".to_string());

    let mut overrides = HashMap::new();
    overrides.insert("port".to_string(), "8443".to_string());

    let merged = merge_operating_parameters(base, overrides);
    assert_eq!(merged.get("host"), Some(&"prod.example.com".to_string()));
    assert_eq!(merged.get("port"), Some(&"8443".to_string()));
    assert_eq!(merged.len(), 2);
}

Optional: collect the keys into a sorted Vec

Add a second assertion: collect merged's keys into a Vec<String>, sort it, and assert it equals vec!["host".to_string(), "port".to_string()]. You'll want .keys(), .cloned(), and .collect() chained together, then .sort() on the resulting Vec (HashMap iteration order isn't guaranteed, so sorting before asserting is the correct habit, not a formality). Why this matters for the backend

Vec, String/&str, and HashMap are the actual data structures nearly every backend-service field is built from — not exotic containers, the same three you've now used directly. Once these three are unremarkable, almost every struct field type in a real codebase reads as "a combination of things I already know," not something new to look up.
