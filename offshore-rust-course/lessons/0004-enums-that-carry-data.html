Lesson 4: Enums That Carry Data

Rust's closest thing to a superpower — and the exact mechanism a backend service like this uses to model errors, outcomes, and state.

Why this matters

Enums-with-data plus exhaustive match are a backend's central modeling tool — its error types, its outcomes, and its state transitions all lean on this one mechanism. If you can read a match block confidently, a large fraction of a real backend stops being intimidating.

Field study first — read a real shape before you write anything

A service like this might model transfer failures with an enum like the one below. Open your editor, paste it into a scratch file (rust-analyzer from Lesson 2 will give you hover on everything), and read it whole before moving on — no need to write the answers down anywhere: 

/// Why a transfer request could not proceed.
enum TransferError {
    OutsideWindow,
    CompensationFaulted(String),
    EscortUnavailable { needed: u32, available: u32 },
}

impl TransferError {
    fn message(&self) -> String {
        match self {
            TransferError::OutsideWindow => "no active access window".to_string(),
            TransferError::CompensationFaulted(reason) => format!("compensation faulted: {reason}"),
            TransferError::EscortUnavailable { needed, available } =>
                format!("need {needed} escorts, only {available} available"),
        }
    }
}

Answer these for yourself before moving on:

  • How many variants does TransferError have, and which one is shaped differently from the other two?
  • message() uses match self { ... }. What does the self pattern in each arm actually bind — the whole enum, or just the data inside one variant?
  • EscortUnavailable { needed, available } is the odd one out. What kind of data does it hold, compared to CompensationFaulted holding a plain String?

The theory

  • A Rust enum is a sum type: a value is exactly one of the listed variants, never more than one, never a mix. Unlike Python's enum.Enum, which is usually just a set of named constants, Rust enum variants can carry their own data — CompensationFaulted(String) is a variant that holds a String, right there in its definition.
  • match is exhaustive: the compiler refuses to build if you forget a variant. In Python, a forgotten elif branch just silently does nothing at runtime. In Rust, a forgotten match arm is a compile error, not a bug waiting to happen in production.
  • Pattern binding: CompensationFaulted(reason) => reason.clone() — the reason in that pattern is bound to the String stored inside that specific variant, and only exists inside that one match arm.
  • One new piece of syntax for the exercise: format!("...{}...", value) builds a String by substituting value into the {} — the same job as Python's f-strings or .format().

    Write the match body — the enum, signature, and test are given

    This is scaffold, not the exercise. Add it to src/lib.rs exactly as shown: 

    /// The result of evaluating a single transfer request.
enum AccessOutcome {
    Ready,
    Blocked(String),
    EscortRequired,
}

/// Returns a one-line human-readable summary of an access outcome.
fn summarize(outcome: &AccessOutcome) -> String {
    todo!()
}

#[test]
fn summarizes_each_outcome() {
    assert_eq!(summarize(&AccessOutcome::Ready), "READY");
    assert_eq!(summarize(&AccessOutcome::Blocked("high seas".to_string())), "BLOCKED: high seas");
    assert_eq!(summarize(&AccessOutcome::EscortRequired), "ESCORT REQUIRED");
}

    One thing in that signature is new and unexplained on purpose: the & in &AccessOutcome means borrow this value rather than take ownership of it — summarize only needs to look at the outcome, not consume it. A full lesson on ownership and borrowing is coming soon; for now, just know that matching on a &AccessOutcome works the same way as matching on a plain AccessOutcome — Rust handles the borrowing inside each match arm for you. (This is exactly the shape message(&self) uses in the TransferError field study above — a method taking &self is this same pattern, just spelled as a method instead of a free function.)

    Write the match outcome { ... } body yourself, handling all three variants, using format! for the Blocked case. Run cargo test summarizes_each_outcome until it's green.

Optional: feel the exhaustiveness check for real

Once green, comment out your EscortRequired match arm and run cargo build. Read the non-exhaustive patterns error rust-analyzer/rustc gives you before uncommenting it. This is the actual compiler guarantee the theory section described — not an abstract claim, a real error you triggered on purpose.

Tangible win A three-variant match you wrote yourself, modeled on the same shape as the TransferError field study above.

Why this matters for the backend

AccessOutcome is a simplified stand-in — a real backend's equivalents (error types, request outcomes, state transitions) follow the identical shape: an enum with differently-shaped variants, and one or more match blocks that handle every one of them. You now know how to read those on sight.

  1. What decides whether the last line of a function's body is its return value, or gets thrown away?
  2. What's the one idea that makes jj's working copy fundamentally different from git's staging area?
Reveal the answers

1. Whether it ends in a semicolon. No semicolon, and it's an expression — its value becomes the function's return value. A trailing semicolon turns it into a statement, and the block's type becomes () instead (Lesson 1).

2. There is no staging area — the working copy is a commit. Every edit is automatically part of the current change; jj describe names it, jj new starts a fresh one (Lesson 3).
