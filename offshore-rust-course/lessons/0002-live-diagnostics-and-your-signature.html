Lesson 2: Live Diagnostics & A Signature You Choose

Understand and verify rust-analyzer's real feedback as you type, then write a function from a description alone — signature and all, not just the body.

Why this matters

Lesson 1 got you syntax colors — that tells you this looks like a keyword. It says nothing about whether your code is actually correct. rust-analyzer is what tells you this is wrong while you're still typing, instead of finding out only when you run cargo build. In a multi-crate backend workspace, that live feedback loop is the difference between confidently editing a module like access_service::eval and getting lost in it.

What rust-analyzer actually adds

Beyond colors, once it's attached to a buffer you get, live, without saving:

  • Diagnostics — type errors, borrow-checker complaints, unused variables — inline, as you type.
  • Hover — the type and doc-comment of whatever's under your cursor.
  • Go to definition / references — jump to where something is defined, or find every place it's used.
  • Code actions — quick fixes it can suggest and apply (e.g. adding a missing use).
  • Rename — rename a symbol everywhere it's used, not just under your cursor.

It also runs clippy for you in the background (confirmed below), so lint warnings show up as diagnostics too — you won't need to remember to run cargo clippy separately most of the time.

Today's tasks

  1. Warm up with a Rustlings rep

    Before writing max below, get if-as-expression under your fingers on its own, with no rust-analyzer verification steps mixed in. Copy just the first file of Rustlings' 03_if set, if1.rs, into rust-practice/src/bin/ and get it green — see the tooling cheatsheet's "Wiring a Rustlings exercise into rust-practice" if this is your first one. For more on the mechanism than task 4 below covers, the Rust Book's "Using if in a let statement" section explains exactly this.

    Optional extra practice: the rest of the 03_if set (if2.rs onward) is good additional reps, not required for this lesson.

  2. Confirm rust-analyzer is attached to your editor

    Once Rust/rust-analyzer support is enabled (Lesson 1), your editor starts an rust-analyzer LSP client automatically whenever you open a .rs file inside a Cargo project — no per-project setup. rust-analyzer itself expects a real binary on $PATH; confirm it's there: 

    rust-analyzer --version

    If this command fails, install it via rustup component add rust-analyzer, or your editor's Rust extension will complain on startup that it can't find the binary — that error message points straight at the fix.

  3. Verify LSP features and clippy-on-save — they're already wired

    Open rust-practice/src/lib.rs and check your editor's LSP status (in Neovim, :LspInfo; in VS Code, the rust-analyzer status item). You should see rust-analyzer listed as attached — no config edit needed to get here. These features work right now for any attached LSP, editor-keybinding differences aside:

    • Hover on the add function name — shows its signature.
    • Go-to-definition on a call to add — jumps to its definition.
    • Find references.
    • Code actions.
    • Rename.

    Clippy-on-save is also worth turning on if it isn't already — most editor integrations expose a "check on save" or "clippy on save" setting for rust-analyzer. Prove it to yourself: temporarily add a line like let _unused = 5; to lib.rs and save — you should see a clippy warning appear as a diagnostic, not just a compiler one. Remove the line once you've seen it.

  4. Verify format-on-save

    Most editor Rust setups can format on save by asking the LSP to format, which shells out to rustfmt under the hood. Enable format-on-save if it isn't already, then prove it: deliberately misformat a line in lib.rs (extra spaces around an operator, a brace on its own line), save the file, and watch it snap back to canonical style with no manual cargo fmt needed.

  5. Vim reminder — before you start this one % / o — confirm the brace match, then open a line below to write the body

    % jumps the cursor to a brace/paren's match — not a mnemonic letter this time, just a motion. While writing a two-branch if/else expression, sitting on the opening { of either branch and hitting % confirms exactly where it closes — handy before you've internalized the shape by feel. o (mnemonic: open a line below) enters insert mode on a fresh line, useful for writing max below the test without manually splitting a line.

    Write a function from a description alone

    One new fact, on top of last lesson's implicit-return rule: if is an expression in Rust, not just a statement. if cond { X } else { Y } evaluates to X or Y — you can use it as a function's return value directly, the same way the last line of a function is its return value.

    This time you get a description and a test — no signature, no stub. That's deliberate scaffold fading, not a stricter version of Lesson 1's exercise: designing a signature is its own skill, separate from filling in a body, and this course starts asking for it early and often from here on. The failing test below is still given to you first, before you design anything against it:

    Write a function that picks the more conservative of two operating limits — e.g. two wind-speed limits, in meters per second — by returning whichever value is lower. 
    #[test]
fn select_conservative_limit_returns_the_lower_value() {
    assert_eq!(select_conservative_limit(12.5, 9.0), 9.0);
    assert_eq!(select_conservative_limit(6.0, 14.0), 6.0);
}

    Add that test to src/lib.rs, then write fn select_conservative_limit(...) above or below it yourself — parameter types, return type, and body, all your call. (One scoping note, not a hint about the logic: the test passes decimal literals, so the types should be a floating-point type, not i32 like last lesson's add.) Use rust-analyzer's hover and inline diagnostics from task 2 while you work — that's what they're for.

Optional: let the compiler catch a real mismatch

Try writing select_conservative_limit with an if/else where one branch ends with a semicolon and the other doesn't. Read what rust-analyzer underlines before fixing it — this is the same "last expression, no semicolon" rule from Lesson 1, just inside a branch instead of a whole function body.

Tangible win cargo test select_conservative_limit_returns_the_lower_value goes green from a function you designed end-to-end — nothing in this lesson told you what the signature should look like.

Why this matters for the backend

Every diagnostic, hover, and go-to-definition you just confirmed working is exactly what you'll lean on reading real backend modules with dozens of types in play — files big enough that you will not want to hold their types in your head unaided.
