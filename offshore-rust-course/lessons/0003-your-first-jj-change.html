Lesson 3: Your First jj Change

jj (Jujutsu) is a good general-practice VCS workflow for managing a sequence of edits. Practice the mechanics now, on real work you've already done.

Why this matters

rust-practice has been sitting there since Lesson 1 as a plain git repo with zero commits — everything you've written is still just untracked files. That's fine for a practice crate, but it means two lessons' worth of real work has never been described anywhere. Many real backend repos run jj colocated with git, so this is a workflow worth having under your fingers early — better to hit the concepts here first, on low-stakes code.

The one idea that's different from git

In git: working directory, staging area (the index), and commits are three separate layers. You git add to stage, then git commit to snapshot.

In jj: there is no staging area, and your working copy is itself a commit — called the working-copy commit. Every file edit you make is automatically part of the current change, live, with no add step. jj describe sets that commit's message. jj new snapshots the current change and starts a fresh, empty one on top when you're ready to move on. There's no separate "finalize this commit" step — describing and moving to a new change are the two actions that do that job.

Today's tasks

  1. Colocate the repo

    From inside rust-practice:

    jj git init --colocate

    This layers jj onto the existing git repo non-destructively — both git and jj commands keep working afterward. This is a common, low-risk way to adopt jj on top of an existing git repo; you're not doing anything practice-crate-specific here.

  2. Look at what jj sees 
    jj st
jj log

    Since the underlying git repo had zero commits, every file you've written — Cargo.toml, Cargo.lock, src/lib.rs, .gitignore — is already part of one working-copy commit. jj log should show that commit marked (no description set). That undescribed commit is Lesson 1's add function and Lesson 2's max function, both still nameless.

  3. Vim reminder — optional, try it once i / A — enter insert mode in $EDITOR for a jj describe message

    jj describe with no -m opens $EDITOR for the message instead — genuinely nicer once a description needs more than one line. It drops you into an editor buffer starting in normal mode, so you need one of vim's insert-mode entry commands: i (insert before the cursor) or A (Append at the end of the line). When you're done, <Esc> back to normal mode, then :wq (write the file, then quit) to save and exit. Same jj outcome as -m "...", different input path.

    Describe it — in your own words, not a template

    Before running the command: without looking back at Lessons 1 or 2, write one sentence for yourself describing what this change actually contains. Not "lesson 1 and 2 work" — what did you actually build? Then turn that sentence into the description: 

    jj describe -m "<your own message here>"

    There's no scaffold to fill in for this one — the content has to come from your own summary of your own work, or it doesn't teach you anything about writing a good change description.

  4. Start a fresh change 
    jj new

    This snapshots the change you just described and gives you a new, empty working-copy commit to work in. Run jj log again — you should now see two commits: your described one, and a new empty (no description set) one on top, ready for Lesson 4.

Recovery, always

If any of this goes sideways — wrong description, jj new when you meant something else — reach for jj undo first. It undoes the last jj operation and is always safe. jj op log shows the full operation history if you need to go back further. Never rm -rf .jj.

Tangible win jj log shows a real, described change instead of "(no description set)" — and the repo now works with both git and jj commands.

Why this matters for the backend

A colocated jj+git setup is a common, low-risk way real backend repos adopt jj — this isn't a practice-crate-only detour. The commands you just ran (jj st, jj log, jj describe, jj new) are exactly what you'll use once you're ready to move from rust-practice to real production changes.

Ask your tutor

If jj git init --colocate errors, or jj log doesn't show what this lesson says it should, stop and ask. jj's error messages are usually good, but "usually good" isn't the same as "obvious on your first day with the tool."
