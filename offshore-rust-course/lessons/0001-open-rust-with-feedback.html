Lesson 1: Toolchain Anatomy, Highlighting, & Your First Function

Understand the five pieces of the Rust toolchain, get nvim to recognize Rust syntax, then write one real line of Rust yourself.

Why this matters

You're going to spend real time reading and writing Rust for an offshore access backend — services built from the crates access-service, telemetry-core, and test-support. Every minute you spend fighting your editor is a minute not spent learning Rust or that backend. Before writing a single line of Rust, it's worth knowing what each tool in the chain actually does — so when something breaks later, you know which piece to blame.

The toolchain, five pieces

People new to Rust often lump these together. They are five separate programs with five separate jobs:

  • rustc — the compiler. Turns .rs files into machine code. You rarely invoke it directly.
  • cargo — the build tool and package manager. The thing you actually type commands into (cargo build, cargo test, ...). Cargo calls rustc for you.
  • rustfmt — the formatter. Rewrites your code to a canonical style. No debates about tabs vs. spaces.
  • clippy — the linter. Catches non-idiomatic or subtly wrong patterns that compile fine but aren't the "Rust way."
  • rust-analyzer — the language server (LSP). The glue that talks to the other four on your behalf and reports back to your editor: live diagnostics, hover docs, go-to-definition, inline type hints.

In Python terms: rustc ≈ the interpreter, cargo ≈ pip + venv + a task runner combined, rustfmt ≈ black, clippy ≈ pylint/ruff, rust-analyzer ≈ pyright/pylance — except in Rust these five are far more tightly integrated, because the compiler's type information is rich enough for the LSP to lean on directly.

Today's tasks

  1. Confirm the toolchain is actually there

    Run each of these and confirm you get a version string back, not a "command not found":

    rustc --version
cargo --version
rustfmt --version
cargo clippy --version

    You should see a version string for all four. This is a quick confirmation, not an install step — if any command isn't found, install the toolchain via rustup first.

  2. Create the practice crate

    This is where every future lesson's exercises will live:

    cd into your projects directory
cargo new --lib rust-practice
cd rust-practice
cargo test

    cargo test should print one passing test — cargo new scaffolds a placeholder it_works test by default. That's your first green run, before you've written a line of Rust yourself.

  3. Open src/lib.rs in your editor, cold

    Open the file in your editor right now, before changing any config. Look at the syntax coloring (or lack of it) and note what you see. This is expected — nothing Rust-specific is wired up yet — but it's worth seeing the "before" state once.

  4. Enable Rust/rust-analyzer support in your editor

    Rust syntax highlighting and language-server support aren't on by default in most editors — enable them once, and every lesson afterward benefits. Most editors offer this through a single built-in setting or a rust-analyzer plugin that wires up treesitter highlighting and the rust-analyzer LSP together. One line of config is usually enough to turn both on.

    Restart your editor and open rust-practice/src/lib.rs. Keywords, strings, and the #[test] attribute should already render in distinct colors.

  5. Vim reminder — before you start this one ci{ — change everything inside the nearest braces

    Vim's editing commands are built from an operator plus a text object, and reading the letters literally tells you what a command does. ci{ is change + inner + the nearest { } pair — "change whatever's inside these braces." With the cursor anywhere inside add's body, ci{ clears todo!() and drops you straight into insert mode, ready to type the real expression — no manual selecting or deleting first. This same operator + text-object pattern reappears constantly (ci(, ci", di{, ...) — once the letters click here, the rest follow the same logic. See ../reference/0004-vim-keybind-cheatsheet.html for the full set.

    Write the body yourself — no copy-paste

    Two Rust facts, and that's all the theory you get before you write real logic:

    • A function is fn name(param: Type, ...) -> ReturnType { ... }.
    • The last expression in a block, with no semicolon, is what the function returns — there's no return keyword required. This is different from Python. Add a semicolon and the expression's value is thrown away instead (it becomes a statement, and the block's type becomes () — "nothing" — instead of the type you meant to return).

    Now do a Rustlings rep on just that mechanic, isolated from the domain code and from the exercise below — the crate you created in task 2 is exactly where it lives. From inside rust-practice: 

    curl -s -o src/bin/functions1.rs \
  https://raw.githubusercontent.com/rust-lang/rustlings/main/exercises/02_functions/functions1.rs

    Open src/bin/functions1.rs, fix it, then run: 

    cargo test --bin functions1

    Green means this warm-up is done — that's the one deliberately-picked file for this lesson. (Full explanation of why the wiring works, and other ways to fetch a file, in the tooling cheatsheet's "Wiring a Rustlings exercise into rust-practice" — this is the pattern every later lesson's Rustlings step reuses without re-explaining it.) For more on how a function is defined than the two facts above cover, the Rust Book's "Functions" section goes deeper.

    Optional extra practice: functions2.rs through functions5.rs (same set, same URL pattern, number swapped) cover the same mechanic from a few more angles. Good reps if you want them — not required to move on.

    Now, the actual lesson exercise. In rust-practice/src/lib.rs, add this exactly as shown — this part is scaffold, not the exercise: 

    /// Returns the sum of `a` and `b`.
fn add(a: i32, b: i32) -> i32 {
    todo!()
}

#[test]
fn add_returns_the_sum() {
    assert_eq!(add(2, 3), 5);
}

    Run cargo test add_returns_the_sum. It fails — todo!() compiles fine but panics the moment it actually runs. That panic is your starting point, not a mistake.

    Now replace todo!() yourself with an expression that returns the sum of a and b, using the implicit-return rule above — no return keyword, no trailing semicolon on that line. Re-run the test until it's green.

Tangible win Rust keywords, strings, and the #[test] attribute render in distinct colors — and you can confirm it's actually treesitter, not a lucky guess, by querying its internal state directly: :lua print(vim.treesitter.highlighter.active[vim.api.nvim_get_current_buf()] ~= nil) should print true. That habit — verify a tool's actual internal state instead of trusting how the screen looks — is worth keeping for every tool you set up in this course.

Optional: break it on purpose (task 5)

Once green, add a semicolon after your expression and run the test again. Read the compiler error (it will mention () and a type mismatch) before removing the semicolon to fix it. This exact error is one of the most common first-week Rust stumbles — better to meet it deliberately now than be confused by it later in a real backend.

Tangible win A red cargo test (from todo!() panicking) turned green from one line you wrote yourself, with no code copied from this lesson. That one line is the entire mission's teaching method in miniature: theory and a failing test are given, the logic is always yours.

Why this matters for the backend

The backend you build toward is a multi-crate Cargo workspace — access-service, telemetry-core, and test-support — same shape as your practice crate, just generalized to more files. Every file you'll eventually open there is a .rs file just like this one — the highlighting, diagnostics, and LSP features you're setting up now are exactly what you'll rely on when reading a real module for the first time.

Ask your tutor

If colors still don't change after restarting nvim, run :checkhealth nvim-treesitter and :TSLog for diagnostics, and if those don't make it obvious — stop and ask rather than guessing further. Same for step 1: if any command doesn't print what this lesson says it should, don't push forward on a broken foundation.
